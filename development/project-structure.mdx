## App directory

The `app/` directory contains all routes using Next.js App Router conventions.

### Locale-based routing

OpenCouncil uses `next-intl` for internationalization. All routes are nested under `[locale]/` to support multiple languages:

```
app/
в”њв”Ђв”Ђ [locale]/
в”‚   в”њв”Ђв”Ђ (city)/[cityId]/           # City-specific routes
в”‚   в”‚   в”њв”Ђв”Ђ (meetings)/[meetingId]/  # Meeting pages
в”‚   в”‚   в”њв”Ђв”Ђ (other)/                 # Non-meeting city pages
в”‚   в”‚   в””в”Ђв”Ђ consultation/            # Consultation system
в”‚   в”њв”Ђв”Ђ (fullscreen)/                # Full-screen layouts (maps)
в”‚   в””в”Ђв”Ђ (interfaces)/                # Special interfaces (chat)
```

### API routes

API endpoints are organized by domain:

<CodeGroup>
```typescript app/api/cities/[cityId]/route.ts
// City CRUD operations
import { getCity, editCity } from '@/lib/db/cities';

export async function GET(request: Request, { params }: { params: { cityId: string } }) {
  const city = await getCity(params.cityId);
  return Response.json(city);
}
```

```typescript app/api/search/route.ts
// Elasticsearch full-text search
import { search } from '@/lib/search';

export async function POST(request: Request) {
  const { query, filters } = await request.json();
  const results = await search(query, filters);
  return Response.json(results);
}
```
</CodeGroup>

### Dev-only components

Components in `components/dev/` must **never** be imported statically in production code:

<CodeGroup>
```typescript Good: Conditional import
const QuickLogin = process.env.NODE_ENV === 'development'
    ? require("@/components/dev/QuickLogin").default
    : null;
```

```typescript Bad: Static import
import QuickLogin from "@/components/dev/QuickLogin"; // вќЊ Don't do this
```
</CodeGroup>

<Warning>
  The condition must use `process.env.NODE_ENV === 'development'` directly (not via a variable) so the bundler can eliminate the dead branch at build time.
</Warning>

## Technology stack

OpenCouncil is built with modern, type-safe technologies:

  <Accordion title="Framework: Next.js 14">

## Import conventions

### Path aliases

Use the `@/` alias for imports from the `src/` directory:

<CodeGroup>
```typescript Good: Using @/ alias
import { getCity } from '@/lib/db/cities';
import { Button } from '@/components/ui/button';
import { auth } from '@/auth';
```

```typescript Bad: Relative imports
import { getCity } from '../../lib/db/cities'; // вќЊ Don't do this
import { Button } from '../components/ui/button'; // вќЊ Don't do this
```
</CodeGroup>

### Import organization

Group imports logically:

```typescript
// 1. React/Next.js imports
import { Suspense } from 'react';
import { notFound } from 'next/navigation';

// 2. Third-party imports
import { Prisma } from '@prisma/client';
import { formatDate } from 'date-fns';

// 3. Local imports (with @/ alias)
import { getCity } from '@/lib/db/cities';
import { Button } from '@/components/ui/button';
import { auth } from '@/auth';
```

<Warning>
  Never use dynamic imports unless explicitly required. All imports should be at the top of the file.
</Warning>

## Code organization principles

### Don't Repeat Yourself (DRY)

<Steps>
  <Step title="Check for duplicated logic">
    If two components have similar code blocks (>10 lines), extract to a shared utility.
  </Step>
  
  <Step title="Common extraction targets">
    - Filtering/sorting logic в†’ Extract to `lib/utils/` or `lib/sorting/`
    - URL parameter handling в†’ Extract to utilities
    - Data transformation logic в†’ Extract to utilities
    - Complex calculations в†’ Extract to helper functions
  </Step>
  
  <Step title="Search before implementing">
    Before adding new logic:
    1. Search for similar code patterns in the codebase
    2. Check if logic exists in multiple places
    3. Extract duplicates to shared utilities
  </Step>
</Steps>

### Component organization

<AccordionGroup>
  <Accordion title="Functional components only">
    - Use functional components with hooks
    - PascalCase for component names
    - camelCase for functions/variables
  </Accordion>
  
  <Accordion title="Server vs Client Components">
    - **Server Components** by default (no `"use client"`)
    - **Client Components** only when needed (`"use client"` directive)
    - Use Server Actions with `"use server"` directive
  </Accordion>
  
  <Accordion title="Type safety">
    - Never cast to `any` or use `any` as a type
    - Use interfaces/types for data structures
    - Proper error typing in catch blocks
  </Accordion>
</AccordionGroup>

## Development environment

### Nix shell

OpenCouncil uses [Nix](https://nixos.org/) to manage development dependencies. All shell commands must be run inside the Nix development shell:

```bash
# Prefix commands with nix develop --command
nix develop --command npm run build
nix develop --command npx tsc --noEmit

# Or open an interactive shell first
nix develop
npm run build  # Now you can run commands directly
```

### Build commands

<CodeGroup>
```bash Development
npm run dev           # Start dev server with Turbo
npm run dev:fast      # Dev server with increased memory
npm run build         # Production build
npm run start         # Start production server
```

```bash Testing
npm test              # Run all tests
npm test -- path/to/file.test.ts  # Run specific test
npm run test:watch    # Run tests in watch mode
npm run test:integration  # Integration tests with testcontainers
```

```bash Database (Prisma)
npm run prisma:generate        # Generate Prisma Client
npm run prisma:migrate         # Run migrations
npm run prisma:studio          # Open Prisma Studio
npm run prisma:migrate:reset   # Reset database
npx prisma db seed             # Seed with sample data
```

```bash Linting
npm run lint          # Run ESLint
```
</CodeGroup>

<Warning>
  When making schema changes, always use `--create-only` to generate the migration file without applying it:
  
  ```bash
  npx prisma migrate dev --name <migration_name> --create-only
  ```
  
  This allows testing the migration against a local database first before applying to production.
</Warning>
